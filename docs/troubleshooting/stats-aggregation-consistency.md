# 전투 기록 기반 통계 집계와 스킬 순서 중복 노출 문제

완성된 통계 화면은 보기에는 단순했지만,
실제로는 "데이터를 어떻게 믿을 수 있게 만들 것인가"가 제일 까다로운 영역이었습니다.

공식 외부 API가 제공되지 않는 환경에서 통계의 신뢰도를 확보하기 위해,
인증된 사용자가 직접 등록한 전투 기록을 기준 데이터로 삼았습니다.
사용자가 서비스를 이용하는 과정에서 자연스럽게 기록이 누적되고,
이를 기반으로 통계가 집계되도록 설계했습니다.

특히, 통계 상세 쪽 스킬 순서 Top N에서 같은 순서가 여러 줄로 나뉘어 노출되는 문제가 있었습니다.

---

## 문제 상황

- 같은 스킬 순서인데도 Top 5나 Top 3에서 승리 기록과 패배 기록이 서로 다른 줄처럼 나뉘어 노출되어, 실제 승률을 한눈에 파악하기 어려웠습니다.
- Top N 정렬이 원본 row 기준으로 먼저 적용되면서, 사용자가 보는 통계 기준과 저장 기준이 어긋났습니다.

처음에는 화면 이슈처럼 보였지만,
실제로는 저장된 스킬 순서를 조회 응답으로 다시 묶어주는 기준이 부족했던 문제에 더 가까웠습니다.

---

## 원인 파악

### 1. 저장 기준과 화면에서 기대하는 집계 기준이 달랐음

`MatchupSkillStat.java`는
방어 스킬 순서와 공격 스킬 순서의 조합을 하나의 row로 저장합니다.
유니크 기준도 `matchup_stat_id`, `defense_skill_order`, `attack_skill_order` 조합입니다.

통계 화면에서는 개별 전투 기록을 매번 다시 조인하기보다
매치업 조합과 스킬 순서별 승패 수치를 빠르게 보여주는 것이 더 중요했습니다.
그래서 통계 테이블은 조회 목적에 맞춰 집계 결과를 문자열 조합으로 보관하는 반정규화 형태로 설계했습니다.

실제 방 상세 조회에서는 수비 슬롯, 공격 기록, 영웅 목록이 각각 도메인 레코드로 남아 있기 때문에
개별 전투 기록을 조회하는 데는 이 구조가 문제가 되지 않았습니다.
문제는 이 반정규화된 집계 row를
공격 스킬 순서 Top N이나 방어 스킬 순서 Top N처럼
한쪽 스킬 순서만 기준으로 보여주는 화면에서 그대로 사용할 때 생겼습니다.

예를 들어 같은 공격 스킬 순서라도 방어 스킬 순서가 다르면
DB에서는 서로 다른 row로 저장됩니다.
그 상태에서 공격 스킬 순서만 Top N으로 보여주면
사용자 눈에는 같은 공격 순서가 여러 줄로 올라온 것처럼 보일 수 있었습니다.

### 2. 스킬 순서 문자열 정규화 기준도 필요했음

스킬 순서는 어떤 스킬을 썼는지뿐 아니라 순서 자체가 의미를 갖습니다.
그래서 별도 테이블로 나누기보다 `"스킬A,스킬B,스킬C"` 같은 문자열 형태로 저장했습니다.

이 방식은 저장과 비교가 단순하다는 장점이 있었지만,
이 값을 다시 Top N 기준으로 사용할 때는
공백이나 구분 형태가 조금만 달라도 DB 입장에서는 다른 값으로 처리될 수 있었습니다.

예를 들면 아래와 같이 공백이 있을 경우 저장 기준으로는 별개가 될 수 있습니다.

- `A1,B2,C3`
- `A1, B2, C3`

### 3. 조회 단계에서 저장된 통계 데이터를 그대로 정렬함

조회 시 저장된 통계 데이터를 스킬 순서 기준으로 다시 합산하지 않고 그대로 정렬해 Top N을 뽑으면,
실제 통계에서는 같은 순서가 여러 줄로 보이게 됩니다.

이 부분은 초기에 설계한 모든 작업들이 끝난 뒤 TOP N 이라는 스킬 순서까지 표시를 하기 위한 새로운 추가 작업을 하며 발생한 반정규화에서 비롯한 문제로,
이 문제는 특히 스킬 순서 통계처럼
"몇 번 나왔는지"를 바로 읽는 화면에서 더 크게 드러났습니다.

---

## 해결 방식

### 1. 스킬 순서는 조회 단계에서 정규화 후 다시 합산

`StatsService.java`에서
`normalizeSkillOrder()`로 공백과 빈 토큰을 정리한 뒤,
정규화된 키를 기준으로 wins / losses를 다시 합산하도록 바꿨습니다.

즉, DB row를 그대로 노출하지 않고,
화면에 올리기 전에 사람이 보는 기준으로 한 번 더 묶었습니다.

여기서 느낀 점은 반정규화를 진행하여 얻는 이점도 있지만, 고도화 작업을 하거나 반정규화와 관련된 데이터를 다루려고 할 때 복잡한 알고리즘 수식이 들어가는 과정을 한 번더 거쳐야 원하는 결과를 얻을 수 있다는 점을 느꼈고, 이 역시 트레이드 오프의 일종이라고 생각합니다.\
처음 설계하던 당시에는 반정규화로 작업을 하면서 얻는 득이 실 보다 많았기에 베스트 프랙티스라 생각을 하였습니다.

### 2. Top N은 합산 결과 기준으로 다시 정렬

`aggregateSkillStats()`에서 정규화 후 합산된 결과를 기준으로
총 경기 수, 승률 순으로 다시 정렬하도록 바꿨습니다.

그래서 같은 스킬 순서가 저장 row 차이 때문에 여러 줄로 보이던 문제를
응답 단계에서 먼저 정리할 수 있었습니다.

여기서 느낀 점도 위의 느낀 점과 비슷한 양상이었습니다.

---

## 결과

- 같은 스킬 순서가 여러 줄로 쪼개져 보이는 문제를 해결했습니다.
- 사용자는 "실제로 많이 나온 순서"를 보게 되고,
  내부 저장 기준 때문에 나뉘어 있던 통계 데이터를 직접 볼 필요가 없어졌습니다.
- 같은 스킬 순서의 승리/패배 수치가 하나로 합산되어, 화면에서 보는 승률도 실제 통계 기준에 더 가까워졌습니다.

이 문제를 정리하면서 얻은 기준은 이거였습니다.

`저장하기 좋은 형태`와 `사용자가 이해하기 좋은 형태`는 다를 수 있습니다.

통계 데이터를 저장할 때는 집계하기 쉬운 구조를 선택하더라도,
사용자에게 보여줄 때는 실제로 비교하려는 기준에 맞게 한 번 더 정리해야 했습니다.
