# 블루/그린 배포 시 EC2 메모리 압박 문제

운영 중 가장 직접적으로 체감된 문제 중 하나였습니다.\
배포가 실패하는 정도가 아니라, 경우에 따라 EC2 자체가 멈춘 것처럼 보일 때가 있었습니다.

---

## 당시 증상

- GitHub Actions 배포가 `Waiting for sena-blue health...` 같은 구간에서 타임아웃 나는 경우가 있었습니다.
- 배포가 길어지는 날에는 EC2가 거의 응답하지 않거나, 재시작 후에야 정상으로 돌아오는 경우도 있었습니다.
- 특히 인스턴스를 껐다 켠 뒤 가동 시간이 어느정도 흐른 뒤에 blue/green 앱이 둘 다 살아 있는 상태가 겹치면 EC2가 응답하지 않았습니다.

이 문제는 단순히 "앱이 느리다"가 아니라,
`redis + mysql + app1 + app2`가 동시에 떠 있는 순간을 t3.small이 버티지 못하는 상황에 가까웠습니다.

---

## 원인 파악

원인을 좁혀 보니 배포 순간에 메모리 피크가 몰리는 구조였습니다.

- MySQL과 Redis는 항상 떠 있음
- 블루/그린 배포에서는 새 앱 컨테이너가 올라올 때 기존 앱도 아직 살아 있음
- 재시작이나 이전 실패 상태가 겹치면 blue와 green이 동시에 남아 있을 수 있음
- nginx 전환 전까지는 두 앱이 잠깐 공존해야 함

결국 평소에는 버티더라도,
배포 순간에만 메모리 압박이 커지는 구조였습니다.

---

## 해결 방식

### 1. 현재 활성 컬러를 기록 파일만 믿지 않고 실제 컨테이너 상태와 함께 판별

`deploy.yml`에서 `active_color` 파일만 기준으로 삼지 않고, 실제로 살아 있는 blue/green 컨테이너를 같이 확인해서 현재 컬러를 다시 잡도록 바꿨습니다.

재시작 이후 상태가 꼬여도 배포 스크립트가 스스로 현재 상태를 다시 해석하게 만든 셈입니다.

### 2. 새 컬러가 뜨기 전에 다음 컨테이너를 먼저 정리

다음 컬러 컨테이너를 띄우기 전에 같은 이름의 컨테이너를 먼저 제거하도록 했습니다.
이 작업이 없으면 실패한 이전 컨테이너가 남아 메모리를 더 잡아먹을 수 있었습니다.\

### 3. 임시 swap을 배포 구간에만 사용

배포 시점의 메모리 피크를 버티기 위해 2GB swap을 임시로 켰습니다.

중요하게 본 기준은 두 가지였습니다.

- 평상시 성능을 위해 swap을 상시 켜두지는 않음
- 배포가 끝나면 `trap`을 통해 정리 시도

즉, 운영 중 상시 성능을 swap에 기대는 구조가 아니라,
blue/green 전환 순간의 완충 장치로만 쓰는 쪽으로 잡았습니다.

### 4. health check와 nginx 전환 순서를 엄격하게 유지

새 컬러 컨테이너가 먼저 정상 응답하는지 확인하고,
그 다음에만 nginx 프록시를 새 포트로 넘기게 했습니다.

전환 실패 시에는 이전 컬러로 nginx를 되돌리고,
승격되지 않은 새 컨테이너도 정리하도록 넣었습니다.

### 5. 전환이 끝난 뒤 이전 컬러를 늦게 종료

프록시 전환 직후에는 기존 연결과 새 연결이 교차하는 짧은 과도기가 있어, 이전 컬러를 즉시 종료하면 일부 요청이 끊길 수 있었습니다. 이를 줄이기 위해 짧은 유예 후 이전 컬러를 종료했습니다.

---

## 결과

- 배포 실패 시 새 컬러 컨테이너가 그대로 남아 다음 배포에 문제를 만드는 상황을 줄일 수 있었습니다.
- 재시작 이후 blue/green 상태가 꼬여도, 배포 스크립트가 실제 실행 컨테이너를 기준으로 다시 판별하게 됐습니다.
- 메모리 여유가 넉넉한 구조는 아니었지만, 최소한 배포 순간에 서버가 통째로 멈출 가능성을 낮추는 방향으로 안정화할 수 있었습니다.

이 문제를 겪으면서 인프라에 대한 지식이 전무한 상황이라 트러블 슈팅 기간이 꽤 길긴 했었지만, 느낀 점은 생각보다 단순했습니다.

소형 인스턴스에서는 평소 리소스보다 `배포 순간의 피크 리소스`를 먼저 봐야 했습니다.
