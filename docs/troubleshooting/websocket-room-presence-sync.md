# WebSocket 접속 상태와 방 참여 상태 싱크 문제

로비에 들어온 사용자의 WebSocket 연결이 살아 있다는 이유만으로
해당 사용자가 참여 중인 방의 멤버 상태까지 `ACTIVE`처럼 동기화되던 문제가 있었습니다.

이 문제는 소켓 연결 여부와 실제로 방에 참여 중인 멤버 상태를 같은 것으로 보면서 생긴 문제였습니다.

---

## 당시 문제

- 로비에만 들어와 있어도 WebSocket 연결 자체는 열려있는 상태였습니다.
- 이 연결 상태가 방 참여자의 표시 상태와 섞이면서, 실제로 방 안에 재입장하지 않았는데도 해당 멤버가 `ACTIVE`처럼 보이는 상황이었습니다.
- 반대로 방 안에 있던 사용자의 WebSocket이 잠깐 끊겼을 때 곧바로 `OFFLINE`으로 확정하면, 일시적인 네트워크 끊김만으로도 온라인/오프라인 표시가 반복해서 바뀔 수 있었습니다.
- 채팅 푸시는 `방에 참여 중이지만 현재 OFFLINE인 멤버`를 기준으로 보내야 했기 때문에, 방 참여 상태와 접속 상태가 섞이면 푸시 대상도 잘못 계산될 수 있었습니다.

정리하자면 `WebSocket 접속 상태`와 `방 멤버의 참여 표시 상태`는 같은 개념이 아니었습니다.

---

## 원인 파악

처음에는 "웹소켓이 붙어 있으면 온라인" 정도로만 보면 될 것 같았는데,
실제로는 아래를 분리해서 봐야 했습니다.

- 접속 상태: 지금 소켓 세션이 살아 있는가
- 방 참여 상태: 이 사용자가 특정 방의 멤버인가
- 방 내부 표시 상태: 현재 `ACTIVE`인지, 잠깐 끊긴 `OFFLINE`인지

특히 길드전 방은 그냥 채팅방이 아니라,
방장 위임, 역할 표시, 예약 상태, 오프라인 푸시 대상까지 전부 `RoomMember` 상태를 기준으로 돌아갔습니다.

그래서 로비 WebSocket 연결만 보고 상태를 맞추면 기준이 틀어질 수밖에 없었습니다.

---

## 해결 방식

### 1. 접속 상태와 도메인 상태를 분리

Redis 기반 WebSocket 세션 관리는 `WebSocketSessionRegistry.java`에서 처리하고,
방 참여 상태는 `RoomMemberService.java`에서 따로 관리하도록 나눴습니다.

즉, WebSocket 쪽은 "지금 연결이 살아 있다"까지만 책임지고,
방 도메인은 "이 멤버를 방 안에서 어떤 상태로 볼 것인가"를 따로 판단하게 했습니다.

### 2. disconnect를 바로 OFFLINE 확정으로 보지 않음

`WebSocketEventListener.java`에서 disconnect가 들어오더라도
바로 `OFFLINE`으로 바꾸지 않고 10초 유예를 둔 뒤 pending 상태로만 등록했습니다.

그 뒤 `WebSocketRoomPresenceScheduler.java`가
3초 간격으로 만료 대상을 확인해서, 정말 재연결이 없을 때만 `markRoomsOffline()`을 호출하도록 했습니다.

### 3. disconnect 이벤트 유실 대비 보정 경로 추가

브라우저 강제 종료나 서버 재시작 뒤에는 disconnect 이벤트를 믿기 어렵습니다.
그래서 `WebSocketActiveMemberReconciliationScheduler.java`에서
30초마다 DB의 `ACTIVE` 멤버와 실제 라이브 세션을 비교해 오래된 `ACTIVE`를 다시 보정하게 만들었습니다.

### 4. 방에 들어온 사람만 방 내부 상태 동기화 대상이 되도록 기준 고정

핵심 기준은 아래와 같습니다.

- 로비에 WebSocket만 연결된 사용자 != 방 참여자
- `ACTIVE / OFFLINE` 전이는 방에 참여한 `RoomMember`에게만 적용
- disconnect는 즉시 퇴장이 아니라 온라인/오프라인 판단을 위한 일시적인 연결 이벤트
- leave / kick만 실제 disconnect 처리

이 기준이 정리되고 나서야 방 내부 사람들에게 보이는 상태와 실제 도메인 상태가 맞기 시작했습니다.

---

## 결과

- 로비 WebSocket 연결만으로 방 참여자의 상태가 `ACTIVE`처럼 바뀌는 문제를 해결할 수 있었습니다.
- 모바일 브라우저나 탭 전환처럼 잠깐 끊기는 상황에서도 멤버 목록의 상태의 문제가 발생하지 않았습니다.
- `ACTIVE`, `OFFLINE`, `leave`, `kick`의 의미가 분리되면서 방장 위임과 푸시 대상 계산도 같이 안정됐습니다.

이 전 다른 프로젝트에서도 실시간 환경을 작업을 했었지만 그 때는 기간이 정해져 있던 프로젝트라 깊게 파고들지는 못 했다면\
이번 프로젝트에서는 시간을 가지고 제대로 실시간 환경을 작업하며 이 문제를 해결하면서 새로운 프로토콜의 실시간 환경에서 상태를 동기화 하는 작업을 SPA 마다 전부 수동적으로 해줘야하던 부분들도 난이도가 높았고\
그 과정에서 인증/인가 파트까지 겹치면서 핸드셰이크를 할 때 어떤 방식으로 데이터들을 끌고올 것인지, 어떻게 토큰들을 관리를 하며 갱신을 할 것인지 또 멤버들의 상태는 어떻게 처리를 할 것인지에 대한 부분 등 복잡한 부분들을 처리하면서 얻어가는 인사이트도 있었습니다.
